## Kontext

Zwei Themen: (1) Publikationscode in Vote- und Result-Screen integrieren. (2) Beim Testen aufgedeckter echter Bug im Rate-Limit der Edge Function.

## Was passiert ist

**PublicationCode-Komponente**
- Neue Komponente `src/components/PublicationCode.svelte` mit Text `CH/OPZL/P/26/0021`
- Positionierung: `position: absolute`, rechter Rand, `bottom: calc(var(--footer-height) * 2)`
- Rotation: `writing-mode: vertical-rl; transform: rotate(180deg)` → liest von unten nach oben (−90°)
- Mobile-Breakpoint: `right: 0.375rem` statt `0.75rem` — verhindert Kleben an Buttons
- Eingebunden in `VoteScreen.svelte` und `ResultScreen.svelte` (je `position: relative` auf `main`)

**Loader-Overlay Layoutshift**
- Problem: `.loader-overlay` hatte `height: 50vh` statt `flex: 1` wie das Options-Grid
- Fix: `flex: 1` → Footer springt beim Vote-Submit nicht mehr

**Edge Function Bug (submit-vote v9)**
- Bug: Live-Vote-Rate-Limit prüfte `created_at` (DB-Insertionszeit), nicht `voted_at` (Cast-Zeit)
- Ursache: Queue-Drain inseriert alte Votes mit `created_at = jetzt` — der nächste Live-Vote sah frisch-inserierten Row und bekam fälschlicherweise 429
- Fix: Live-Vote-Check auf `voted_at >= (now - 15s)` geändert
- Deployed als v9 via Supabase MCP

**Prozess-Notiz**
- Bug wurde zunächst zweimal falsch eingeschätzt («kein Bug, Rate Limit korrekt») — erst nach Lesen des Edge-Function-Codes klar. Feedback gespeichert.

## Entscheidungen

- **PublicationCode als eigene Komponente** — einmalige Änderung, zentrale Stelle für Text + Stil
- **writing-mode statt transform: rotate** — vermeidet Overflow-Positioning-Komplexität
- **voted_at statt created_at für Rate Limit** — deckt Queue-Drain-Szenario korrekt ab; `created_at` war strukturell falsch für diesen Use Case
